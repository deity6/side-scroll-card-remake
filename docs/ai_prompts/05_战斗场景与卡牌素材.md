# 战斗场景与卡牌素材规范

## 当前版本

- 版本：v0.3.2
- 主题：手绘简约萌宠 + 森林冒险 + 春机盎然
- 目标：战斗场景、玩家 HUD、手牌和 battle card 使用统一的萌宠森林风格，并保持可维护的节点层级和资源命名。

## 正式资源目录

- 战斗场景 UI：`assets/ui/battle_scene_pet_spring/`
- 战斗卡牌：`assets/cards/battle_card_pet/`
- AI 制作过程：`assets/ai制作/`

正式场景只引用 `assets/ui/` 与 `assets/cards/` 下的稳定命名资源。`assets/ai制作/` 用于保留过程稿、透明图和备用素材，不作为运行时首选路径。

## Battle Scene 节点原则

- `battle_scene.tscn` 保留现有战斗逻辑节点名，避免破坏 `battle_scene.gd` 的 `%NodeName` 引用。
- 视觉贴图节点与逻辑节点分离，例如按钮贴图使用 `TextureRect`，点击逻辑仍由透明 `Button` 承接。
- 玩家 HUD 区域使用 `Control` 手动布局，避免 `Container` 抢占子节点位置。
- 动态列表仍可保留容器，例如 `EquipContainer` 继续使用 `HBoxContainer`。

## z-index 规划

总层数控制在 `0..14`：

| 层级 | 用途 |
| --- | --- |
| 0 | 背景与默认贴图 |
| 1 | 敌人区域；敌人影子局部 0、敌人立绘局部 1 |
| 2 | 玩家底部大背景 |
| 4..13 | 正常手牌堆叠，最多 10 张手牌 |
| 14 | 飞入动画、选中、拖拽、Debug 和 ResultPanel 前景 |

卡牌内部不使用高 z-index。卡牌内部依靠节点顺序渲染，整张卡作为一个独立渲染单元参与手牌堆叠。

## Battle Card 分层

- 标准尺寸：`240x360`
- 根节点：`BattleCard`，单张卡牌的整体控制节点，手牌堆叠时应作为一个完整渲染单元。
- 关键节点：
  - `CardBase`：卡牌底板贴图，按卡牌类型切换基础颜色与质感。
  - `CardFrame`：卡牌外框与插画窗口边框，覆盖在底板之上。
  - `CardName/CardNameLabel`：卡牌名称区域与名称文本。
  - `CardDescLabel`：卡牌效果描述文本，使用固定区域裁剪避免越界。
  - `TypeLabelArea/TypeLabel`：卡牌类型标签区域与类型文字。
  - `CostArea/CostIcon/CostValue`：费用显示区域、费用图标与费用数字。
  - `Highlight`：选中或可交互时的高亮提示层。
  - `ShadowOverlay`：禁用、不可出牌或遮罩状态层。

`battle_card.gd` 根据卡牌 `type` 切换底板和边框：

- `attack`：攻击牌底板与攻击框
- `action`：行动牌底板与行动框
- `equip`：装备牌底板与行动框备用

## 手牌动画

- 正常堆叠：`_card_stack_z(index) = index`
- 飞入前景：`_card_entry_z(index) = 10`
- `CardArea` 的 z-index 为 4，因此正常手牌实际渲染层为 4 到 13，飞入/选中/拖拽实际层为 14。
- 飞入动画结束后，卡牌必须回到正常堆叠层。

## 注意事项

- 不要使用 `1000`、`10000` 这类大 z-index。Godot `CanvasItem.z_index` 有上限，且过大的层级会降低可维护性。
- 如果卡牌内部文字遮挡相邻卡牌，优先检查卡牌内部是否设置了高 z-index，而不是继续增加父卡牌 z-index。
- `RichTextLabel` 用于卡牌文字时，需关闭 `fit_content`，启用固定区域裁剪，避免文本越界。
